     Hero Redesign — Spatial Image-Mask Reveal

     Context

     The current hero has one Pexels image at a time rotating every 6 seconds,
     with a circular cursor mask revealing it inside the headline text via
     background-clip: text. The user wants something more sophisticated:
     multiple images positioned spatially across the headline, so that as the
     cursor moves left-to-right, different images are revealed depending on
     cursor position. The mask circle still cuts the reveal so only pixels near
     the cursor are shown — but now what's revealed depends on where you are.

     Separately: the italic descenders in line 2 ("people actually use.") are
     getting clipped at the bottom — p, y, . go below baseline but .italic-line
     has overflow: hidden with only padding-bottom: 0.08em, which at 168px font
     size is ~13px and not enough for Fraunces italic descenders.

     How the effect works (technical approach)

     Single element (.headline.overlay) with CSS multi-background:

     .headline.overlay {
       background-image: url(img1), url(img2), url(img3), url(img4);
       background-size: 25% 100%;
       background-position: 0% 0%, 33.33% 0%, 66.67% 0%, 100% 0%;
       background-repeat: no-repeat;
       background-clip: text;
       color: transparent;
     }

     Each image occupies one 25%-wide horizontal stripe across the full headline
      height (covering both lines). Because background-clip: text clips each
     layer to the text shape, the images appear inside the letters of whichever
     region they cover.

     The existing radial mask on .overlay-mask (parent) reveals only a circle
     around the cursor — that already works. Combined with the spatial image
     layout, this gives the user-described effect: cursor at left reveals image
     1; as you sweep right the revealed image transitions through 2 → 3 → 4. No
     JS image-cycling needed; spatial position alone drives which image you see.

     Why this approach over alternatives:
     - Canvas/SVG approach would work but adds runtime complexity and
     font-loading races for text geometry
     - Stacked-divs approach (4 copies of the headline each clip-pathed to its
     column) duplicates DOM and is fragile when fonts/text reflow
     - Multi-background is one element, zero JS for the spatial logic,
     GPU-composited, and keeps text geometry as the single source of truth

     Implementation

     1. Curate 4 images (use existing Pexels set, pick the most visually 
     distinct 4)

     Keep them defined in Hero.tsx for now. Preload via new Image() on mount as
     today.

     2. Update .headline.overlay in globals.css

     Replace the single-image rules (background-size: cover; 
     background-position: center;) with the multi-background layout:
     - background-size: 25% 100%
     - background-position: 0% 0%, 33.33% 0%, 66.67% 0%, 100% 0%
     - background-repeat: no-repeat
     - Drop transition: background-image 0.6s ease (no longer rotating a single
     image)
     - Keep background-color: var(--lime) as fallback before images load

⏺  Build clean. Here's what was wrong and what's fixed:

  Root cause: CSS custom properties used directly inside mask-image: 
  radial-gradient(circle at var(--mx) var(--my), …) are treated as untyped
  tokens by the browser. Many engines won't trigger a mask repaint when those
  vars change via JS unless they're registered with @property and a concrete
  syntax type. The original HTML may have worked because of single-render
  context, but in our React/Lenis/Turbopack pipeline the repaint never fires.

  Fixes:
  1. Added @property --mx and @property --my with syntax: "<length>" and
  initial-value: -9999px in globals.css — now the browser knows these are
  lengths and recomputes the gradient on every change.
  2. Switched the mousemove handler in Hero.tsx to gsap.quickSetter + a single
  requestAnimationFrame batcher — one var write per frame, no thrashing,
  GSAP-managed unit handling.
